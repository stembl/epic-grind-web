Join the Build

This walks you through going from zero to making real contributions to Epic Grind. No prior dev experience required. You'll be writing prompts, not code.

~30 min setup Windows or Mac Cursor AI included
1 Create a GitHub Account

GitHub is where the code lives. You need a free account to get access to the project.

Do this

Go to github.com/signup and create a free account. Pick a username you're comfortable with — it'll appear in commits.

Once your account is set up, send Steve your GitHub username or the email address you signed up with. He'll add you as a collaborator on the repository.

Send Steve

Your GitHub username (e.g. john-doe) or the email you used to sign up. Steve will invite you to the repo — you'll get an email from GitHub to accept.

Accept the invite before moving to the next step. The invite email comes from noreply@github.com and expires after 7 days.

2 Install Cursor

Cursor is a code editor with a built-in AI agent. You use it to explore the project, understand what everything does, and make changes by describing what you want in plain English.

Do this

Go to cursor.com, download the installer for your OS (Windows or Mac), and run it. It installs like any other app.

On first launch, Cursor will ask you to sign in or create an account. Use your email. The free tier is sufficient to get started.

Note

Cursor is built on VS Code. If you've used VS Code before, it's identical except for the AI panel on the right side.

3 Clone the Repository

"Cloning" downloads a local copy of the project to your machine so you can work on it. Changes you make stay local until you intentionally push them back up.

Do this

Open Cursor. Press Ctrl+Shift+P (Windows) or Cmd+Shift+P (Mac) to open the command palette. Type Git: Clone and hit Enter.

When prompted for a URL, paste:

https://github.com/stembl/epic-grind.git

Choose a folder on your machine to save it to (e.g. C:\work or ~/projects). Cursor will download the project and ask if you want to open the folder — say yes.

Alternative — Terminal

Open Cursor's built-in terminal with Ctrl+` and run:

$ git clone https://github.com/stembl/epic-grind.git $ cd epic-grind
4 Learn the Project with Cursor AI

Before changing anything, spend 15 minutes asking the AI to orient you. Open the chat panel (Ctrl+L or click the chat icon on the right sidebar) and try these prompts:

Agent prompt — Project overview

"Walk me through the project structure. What does each top-level folder contain and what is its purpose?"

Agent prompt — Understand the game loop

"Explain how quests work end-to-end. Starting from when the user taps a quest card, what code runs and in what order?"

Agent prompt — Find where things live

"Where is all the story and quest text written? Show me an example quest and explain how the prose variables work."

Agent prompt — Understand the data layer

"How does the app save user progress? What goes to Supabase and what stays local?"

The AI has full context of the codebase. Ask it anything — there are no dumb questions here. If an answer doesn't make sense, ask it to explain differently or give an example.

5 Make Changes Safely

Never work directly on main. Always create a branch first. A branch is an isolated copy of the code where your changes can't break anything until they're reviewed.

Open the terminal (Ctrl+`) and create a branch with a name that describes what you're doing:

$ git checkout -b your-name/what-youre-changing # examples: $ git checkout -b alex/week3-quest-prose $ git checkout -b alex/fix-xp-display

Now describe your change to the AI agent and let it make the edits:

Agent prompt — Before making any change

"I want to [describe what you want to change]. What files would be affected and what's the safest way to do this?"

Agent prompt — Making the change

"Update the assignment prose for personal_day5 to feel more urgent and raise the tension. Keep the same voice as the surrounding quests."

Agent prompt — Review before committing

"Review the changes you just made. Are there any issues, typos, or things that could break? Check for consistency with the rest of the file."

Rule of thumb

If you're not sure whether a change is safe, ask the agent: "What would break if I changed this?" before touching it.

6 Commit and Push

Once your changes look right, save them to Git and push them up so Steve can review.

Check what changed:

$ git status $ git diff

Stage and commit:

$ git add -A $ git commit -m "Brief description of what you changed" # examples: # git commit -m "Update day 5 assignment prose to raise tension" # git commit -m "Fix typo in Orin dialogue on calibration screen"

Push your branch:

$ git push origin your-branch-name

Then open a Pull Request on GitHub so Steve can review and merge it:

Do this

Go to github.com/stembl/epic-grind. GitHub will show a yellow banner saying your branch was recently pushed — click Compare & pull request. Add a short description of what you changed and why, then click Create pull request.

Agent prompt — Write your PR description

"Help me write a pull request description for the changes I just made. Keep it short — what changed and why."

7 Staying Up to Date

Other people (and the main dev) push changes regularly. Keep your local copy current before starting new work:

$ git checkout main $ git pull origin main

Then create a new branch from the updated main for your next change. Never work directly on main.

If you get a merge conflict

This happens when two people changed the same file. Don't panic — ask the agent: "I have a merge conflict in [filename]. Walk me through resolving it safely."

Part 2 — Testing on Android

These steps set up a live Android testing environment identical to the main dev machine. You run the app on a real phone, make a change, and see it update in seconds.

8 Install Flutter

Flutter is the framework the app is built in. You need it installed locally to run and rebuild the app. The agent handles the commands — you just need the tools present.

Do this

Open Cursor with the epic-grind project folder open. Open the chat panel (Ctrl+L) and paste this prompt exactly:

Agent prompt — Install Flutter

"Check whether Flutter is installed on this machine by running 'flutter --version' in the terminal. If it is not installed, give me the exact steps to install it for my OS, including adding it to PATH. Do not guess — run the check first and report what you find."

The agent will detect your OS and walk you through installation. Follow each step it gives you, then run the check again at the end to confirm it worked.

Windows note

Flutter requires adding itself to your system PATH. The agent will show you exactly how. After updating PATH, close and reopen the terminal before running any flutter commands.

Once Flutter is confirmed installed, run this to install the project's dependencies:

Agent prompt — Install dependencies

"Run 'flutter pub get' in the terminal from the epic-grind project root. Report any errors."

9 Connect Your Android Device

You need a physical Android phone or a computer fast enough to run an emulator. A real phone is easier and gives you a better read on how the app actually feels.

On your Android phone:

Enable Developer Mode

Go to Settings → About phone. Tap Build number seven times in a row. You'll see "You are now a developer." Go back to Settings — a new Developer options menu has appeared.

Enable USB Debugging

Inside Developer options, turn on USB debugging. Plug your phone into your computer with a USB cable. A prompt will appear on the phone asking to trust this computer — tap Always allow.

Now confirm the device is visible:

Agent prompt — Confirm device detected

"Run 'flutter devices' in the terminal and tell me what devices are listed. If my Android phone is not showing, tell me what might be wrong and how to fix it without guessing — run diagnostics first."

Using an emulator instead

If you don't have a physical device, ask the agent: "Set up an Android emulator on this machine using Android Studio's AVD Manager. Walk me through it step by step."

10 Run the App on Device

This launches the app directly onto your connected phone. The first build takes 2–5 minutes. After that, changes update in seconds via hot reload.

Agent prompt — Launch app

"Run the app on my connected Android device using 'flutter run'. If there is more than one device available, ask me which one to use before running. Report any build errors in full."

The app will install and open on your phone. The terminal stays active — keep it open.

While the app is running, these keyboard shortcuts work in the terminal:

r — hot reload (apply code changes instantly, keeps app state) R — hot restart (full restart, resets state) q — quit
If the build fails

Copy the full error text from the terminal and paste it to the agent: "Flutter build failed with this error. Diagnose and fix it: [paste error]"

11 Make a Change and See It Live

This is the core loop: describe what you want changed → agent edits the code → hot reload → see it on your phone. No build step needed for most changes.

Agent prompt — Change quest prose

"In quests.dart, find the assignment text for personal_day3. Rewrite it to open with immediate action — something is already wrong when the user arrives. Keep it under 60 words. Do not change any other fields in that quest."

Agent prompt — Change a UI label

"Find where the 'Move out' button label is defined in quest_screen.dart and change it to 'Into the breach'. Only change the label text, nothing else."

Agent prompt — Add a story day

"In quests.dart, show me the structure of an existing story-type quest so I can see the exact format. Then I'll tell you what to add."

After the agent makes changes, go to your terminal and press r to hot reload. The change appears on the phone within 2 seconds. No reinstall required.

When hot reload isn't enough

Some changes (new screens, new data structures, navigation changes) require a full restart. Press R in the terminal instead of r.

12 Build a Shareable APK

When you've tested a set of changes and they're solid, build a release APK that can be installed on any Android device without a cable. This is what gets shared as a tester build.

Agent prompt — Build release APK

"Build a release APK using 'flutter build apk --release'. Run it from the epic-grind project root. When it finishes, tell me the exact file path of the output APK and its file size."

The APK will be at build/app/outputs/flutter-apk/app-release.apk inside the project folder. Build takes 3–8 minutes the first time.

Share the build

Send the APK file to Steve, or follow the release process on the GitHub repo to attach it to a release. Do not share release APKs publicly without Steve's sign-off.

Agent prompt — Full test cycle after changes

"I've made changes to [list what you changed]. Before I build the APK, check the affected files for any obvious errors, missing required fields, or inconsistencies with the surrounding code. Report findings — do not fix anything yet."

13 The Full Change Loop

Every session follows the same pattern. Once it's muscle memory, each cycle takes minutes.

# 1. Get current with main git checkout main git pull origin main # 2. Create a branch for your work git checkout -b your-name/what-youre-changing # 3. Launch app on device (keep terminal open) flutter run # 4. Describe changes to agent → press r to reload → test on phone # 5. Repeat until it's right # 6. Review before committing # Ask agent: "Review everything I changed this session." # 7. Commit and push git add -A git commit -m "description of what changed" git push origin your-branch-name # 8. Open a pull request on github.com/stembl/epic-grind
If you break something

If the app stops launching or something breaks unexpectedly, ask the agent: "Show me every file I've changed since my last commit using 'git diff'. Walk me through what changed and identify what's most likely broken." You can also run git stash to temporarily revert all changes and confirm the app runs clean.

Quick Reference — Agent Prompts
Understand any file

"Explain what this file does and how it connects to the rest of the app."

Find where something is

"Where is the text for [screen/feature] written? Show me the relevant code."

Propose a change safely

"I want to change [X]. What's the safest approach and what could break?"

Review your work

"Review the changes I just made. Any issues, inconsistencies, or things to watch out for?"

Fix something broken

"The app is showing [describe the problem]. What's causing it and how do I fix it?"

When you're stuck

"I'm trying to [goal] but I'm not sure where to start. Walk me through it step by step."

Android Testing Prompts
Check environment

"Run 'flutter doctor' and explain every warning or error in plain language. Tell me what to fix and what can be ignored."

Device not showing up

"Run 'flutter devices' and 'adb devices'. My phone is plugged in but not appearing — diagnose why without guessing."

Build error

"Flutter build failed. Here is the full error output: [paste]. Identify the root cause and fix it."

Verify changes before building

"Show me every file I've changed since my last commit using 'git diff --stat'. Walk me through each change and confirm nothing looks unintentional."

Runtime crash

"The app crashed on my phone. Show me the last 50 lines of flutter output from the terminal and tell me what caused it."
